# Zen of AI Coding

## Summary
这不是一篇学术论文，而是一篇关于“AI 代理重塑软件工程”的观点文章。作者主张代码生成成本正在快速趋近于零，开发者的核心价值将从“写代码”转向“定义问题、设置约束、建立反馈与防护栏”。

## Problem
- 文章要解决的问题是：在编码代理快速变强的背景下，软件团队应如何重构开发流程，才能**更快交付且不牺牲可靠性与安全性**。
- 这之所以重要，是因为当“写代码”不再是主要瓶颈后，真正的瓶颈转向了**需求澄清、测试验证、安全审查、发布流程、运维风险和人类认知协调**。
- 作者还指出，如果缺少反馈回路与治理机制，代理只会把低质量实现更快地产出，导致“fast rubbish is still rubbish”。

## Approach
- 核心机制很简单：把 AI 代理视为**高吞吐执行层**，而把人的角色升级为**问题定义者、上下文设计者、约束设定者和结果裁判者**。
- 通过建立**紧密反馈回路**来驱动代理迭代，包括测试、CI、日志、UI 检查、可观测性与验收标准，使代理不断朝“可验证正确”收敛，而不是生成“看似合理的胡说”。
- 采用**agent-first 软件工程**：不仅让代理写代码，还让其参与业务分析、运维、迁移、质量保证等任务；同时对权限、审计、隔离和回滚进行严格设计。
- 采用**agent-first 产品策略**：服务和 API 不应只为人类设计，而应暴露结构化、机器可读的接口，让代理成为一等用户。
- 在组织层面，作者建议利用“代码便宜、重构便宜、技术债更易偿还”的新经济学，多做小实验、快速重构，但前提是加强 guardrails 与 failure-mode 预案。

## Results
- 文中**没有提供系统化实验、基准数据集或正式定量评测**，因此不存在可严格复现的学术指标结果。
- 最具体的案例声明之一是：作者将**4 次**从零重建 CMS、每次采用不同架构，以更快学习需求与架构取舍；但未提供时间、成本或性能对比数据。
- 另一个具体案例是：作者称使用 Claude 将**4 个 WordPress 博客**从旧主机迁移到 Hetzner，耗时约**15 分钟**；但未给出人工基线、错误率或迁移成功率。
- 作者还声称其团队构建过可**一次性生成完整营销网站**的代理，以及“几分钟而不是几个月”生成多语言营销网站的能力，但没有给出转化率、开发工时节省比例或质量评估。
- 最强的可操作性主张不是性能突破，而是流程性结论：在代理时代，最高杠杆投资应从“多写代码”转向**测试、评测、可观测性、权限控制、回滚机制和面向代理的产品接口**。
- 文章的主要贡献更接近**工程范式总结与实践宣言**，而非提出一个经过验证的新算法或新模型。
