# A Practical Guide for Establishing a Technical Debt Management Process (Preprint)

## Summary
本文提出一套可落地的技术债管理（TDM）建立指南，基于跨3个团队的行动研究复现，提炼出共通做法、常见挑战与实践建议。其价值在于把分散的技术债研究方法转化为团队可直接采用的流程模板。

## Problem
- 软件团队常会采用短期有利但长期阻碍演进的方案，即技术债；若缺少系统化管理，后续变更成本、风险与维护负担会持续累积。
- 现有技术债管理研究很多，但实践迁移不足，缺少可跨公司复用的“标准式”建立流程与最佳实践。
- 团队尤其难回答：**该采用哪些TDM方法、如何落地、会遇到哪些共性障碍、以及流程是否真的提升了技术债意识。**

## Approach
- 作者复现其先前提出的**5步工作坊方法**，在3个团队中开展**行动研究**，让团队从已有研究方法中选择并试用适合自己的TDM做法。
- 研究覆盖**15次工作坊 + 4次回顾**，并长期旁听团队会议，记录他们如何识别、记录、排序、偿还和预防技术债。
- 为比较可迁移做法，作者分析各团队最终采用方案的**共性与差异**，提炼所有团队都接受的起点性“最佳实践”。
- 为评估意识变化，作者结合**系统观察、问卷、TD-SAGAT中断式感知测量**，检查团队在决策时是否会考虑替代方案、收益、缺点、风险、利息、受影响组件和质量属性等因素。
- 最终产出包括一份面向从业者的**白皮书式指南**，以及对缺陷跟踪/议题系统供应商的工具支持建议。

## Results
- 数据规模：覆盖**3个团队、20名参与者、108场会议、96小时观察、30个月净研究周期**；共分析**1132个讨论议题**，并在**28场会议**中收集**240份TD-SAGAT数据**。
- 所有团队都采用了若干共通做法：**设立TD champion/manager**、在**现有backlog中新增TD项类型**、记录相似属性（如**interest、contagiousness、resubmission date、提醒讨论缺点与风险**）、并将这些做法作为TDM最佳实践起点。
- 作者声称跨团队稳定出现的核心实践包括：**(1) TD负责人；(2) 带具体取值的TD记录属性；(3) 用“备选方案+缺点+风险”来去偏决策；(4) TD优先级主要与TD相互比较，而非直接与功能需求比较；(5) 使用resubmission date进行回看。**
- 团队间也存在差异：**优先级方法与偿还方式并不统一**，说明这些部分更依赖具体团队情境，难以形成单一标准流程。
- 常见挑战包括：需要**更具体的操作支持**、需要**简单方法从backlog中识别TD**、需要**如何启动流程的指导**、在早期需要**外部提醒以坚持流程**、以及需要帮助团队**表述TD后果**。
- 关于效果，论文给出的最强定量结论较有限；其明确声称：**总体TD意识只在议题类型中加入相关属性并形成持续提醒时才会提升**。文中摘录未提供更细的统一数值提升幅度或与基线/对照组的显著性比较。
