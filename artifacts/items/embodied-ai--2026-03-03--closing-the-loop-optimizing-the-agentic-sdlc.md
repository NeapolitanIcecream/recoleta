# Closing the Loop – Optimizing the Agentic SDLC

## Summary
这是一篇关于**代理式软件开发生命周期（agentic SDLC）优化**的实践性文章，核心目标是让编码代理不仅能写代码，还能独立完成运行、测试、调试与验证，从而真正“闭环”。文章提出一套围绕 worktree、端口管理、守护式服务、日志回流和浏览器测试的工程工作流。

## Problem
- 文章要解决的问题是：**代码生成已经很便宜，但代码评审、运行验证、测试与监控成了新的瓶颈**，尤其是在多个代理并行开发多个功能时更明显。
- 多代理直接共享同一代码仓库会导致**相互覆盖修改、环境互相干扰、端口冲突、调试困难**，使得并行开发失控。
- 如果代理不能自己运行应用、读取日志、驱动浏览器验证结果，人类就必须手工补上这些反馈回路，**无法规模化地“让代理交付软件”**。

## Approach
- 用 **git worktrees** 给每个代理提供隔离的代码副本与运行环境，避免同仓并发编辑造成冲突，同时将日志和运行产物按 worktree 隔离存放。
- 用**分支名哈希得到稳定端口**，避免多个 worktree 默认争抢同一端口；并在 worktree 根目录写入 `.dev/manifest.json`，记录端口、PID、时间戳，让其他工具和代理能发现运行实例。
- 将开发服务器改造成**幂等的守护式接口**：代理不直接启动服务，而是调用 `dev:up / dev:status / dev:down`；如果服务已存在则无操作，从而避免重复启动、误杀或状态误判。
- 将**应用日志路由回 worktree 内部**，并明确告诉代理日志位置，让它在调试时能查看真实运行错误、性能与异步任务结果，而不是只靠静态猜测源码。
- 给代理接入**浏览器自动化测试能力**，教它如何登录和使用应用，并要求其收集截图/视频等证据、自我审查结果，必要时还能用子代理拆分“写测试标准”和“执行测试”两个任务。

## Results
- 文中**没有提供正式实验数据、基准数据集或可复现实验指标**，因此没有可量化的 SOTA 或 benchmark 对比结果。
- 最明确的定量表述是作者声称自己每天需要处理 **20K+ LOC** 的代理改动，这成为推动代理自测与闭环验证的直接动机，但不是严格实验结果。
- 文章的核心主张是：通过 worktree 隔离、稳定端口、幂等服务、日志回流和浏览器测试，能把代理从“只会写代码的自动补全”提升为**能独立完成验证与交付的协作同事**。
- 作者还声称这种方式可以支撑“**10+ terminals**”式的多代理并行工作流，让并行开发从混乱变成可管理的杠杆。
- 另一个具体收益是：代理可以在本地测试时自动产出**截图和视频证据**，并可附到 PR 中，从而减少人工验收负担；但文中未给出节省时间或缺陷率下降的数字。
