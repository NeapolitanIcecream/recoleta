# Show HN: A governance pattern for self-evolving AI skills

## Summary
这项工作提出一种用于 Claude Code Skills 的“自进化技能”治理模式，使技能在真实使用中逐步积累可复用知识，同时通过严格门控避免上下文膨胀与知识退化。其核心贡献是把自进化从模型/代理层下放到提示注入与外部知识库层，在不改模型权重的前提下实现可控演化。

## Problem
- 传统 Skill 是静态的：作者写一次、用户反复调用，但跨会话发现的领域知识不会沉淀，导致每次都“从零开始”。
- 在数据库排障、代码库分析、业务系统集成等场景中，AI 会不断发现稳定且可复用的知识；若不保存，会浪费推理成本与上下文窗口。
- 直接把所有历史知识都塞进提示会造成噪声、冗余和上下文膨胀，还可能引入错误累积，因此需要有治理约束的记忆增长机制。

## Approach
- 核心机制是一个**五重门（Five-Gate）治理协议**：只允许满足“可复用、与现有知识一致、不冗余、具备时效标记、放到正确位置”的知识写入，且常见结果是“不写入”。
- 知识以 `references/` 下的**活知识库**形式维护，而不是改模型参数；典型文件包括 `_index.md`、`schema_map.md`、`query_patterns.md`、`business_rules.md`、`investigation_flows.md`。
- 采用**按路由选择性加载**：先读 `_index.md`，再只加载当前任务相关的 1–2 个主题文件，而非全量注入，从而控制上下文开销并避免知识库越长越重。
- 设计上强调**需求驱动、选择性增长、成熟后趋于稳定**：不做自主探索、不做脚本自修改、不追求永久增长，而是让知识在业务变化前自然收敛。
- 该模式与工具层解耦：可用 Python 脚本，也可替换成 MCP；真正关键的是治理协议、知识组织方式与选择性注入机制。

## Results
- 文中**没有给出标准化定量指标**（如准确率、成功率、token 节省百分比、与基线的数值对比），作者明确表示“不需要 KPI 式指标”，主要依赖定性成熟度信号评估。
- 给出了明确的**上下文控制阈值**：当知识库增长到 **5 个主题文件、每个 50–80 行** 时，若全量加载会浪费约 **250–400 行**上下文；因此要求 `_index.md` 保持在 **40 行以内**，单主题文件超过 **~80 行**则拆分，总主题文件超过 **8** 个时审查合并机会。
- 声称已在**真实数据库**上进行了“完整演化实验”，并保留了**全量演化日志、Five-Gate 决策记录、质量审计、逐轮知识快照**，可对比任意两轮观察知识如何增长，但摘录中未报告具体实验集、轮次统计或性能数字。
- 最强的具体主张是：该模式把自演化限制在**上下文层**，因此即使知识库出错，最坏情况也只是“一个错误参考事实”，而不是模型层面的**不可逆行为改变**，从而相较参数级自演化具有更高安全性。
- 另一项具体主张是：当领域知识覆盖 **90%+ 日常场景** 时，技能应进入“成熟/稳定”阶段，只在业务变更（如新表、新规则、新流程）时恢复增长；这是其应对稳定性—可塑性矛盾的实践原则，而非量化实证结果。
