Summary

这项工作提出了一种用于 Claude Code Skills 的“自演化技能”治理模式，让技能在真实使用中积累可复用知识，同时通过严格门控避免上下文膨胀与知识退化。其核心贡献是把自演化代理思想下沉到Skill/提示注入层，在不改模型参数的前提下实现可控演化。

Problem

  • 传统 Skill 是静态的：作者一次编写、用户反复调用，但在数据库排查、代码库分析、业务集成等场景中，AI 在使用过程中发现的结构关系、查询模式、业务规则无法沉淀，导致每次会话都“从零开始”。
  • 若直接持续追加知识，知识库会迅速变成噪声，既降低准确性，也浪费有限的上下文窗口。
  • 需要一种可增长但可收敛的机制：既能跨会话复用经验，又能避免无限增长、冲突累积和失控演化，这对代码智能、软件运维与复杂业务系统分析都很重要。

Approach

  • 核心机制是一个面向 Skill 的活知识库：在 references/ 中按主题维护可复用知识，如结构映射、查询模板、业务规则、排查流程，而不是把 Skill 文档固定死。
  • 通过 Five-Gate Governance Protocol 决定是否写入知识：检查是否有复用价值、是否与已有知识冲突、是否冗余、是否时效敏感、以及应放在哪个文件；最常见结果是“不写入”，以此抑制噪声增长。
  • 通过 _index.md 做路由式按需加载：先读一个小型路由表，再只加载当前任务相关的 1–2 个主题文件，而不是把全部知识注入上下文，从而控制上下文开销。
  • 采用需求驱动、选择性增长、成熟后稳定的演化哲学：只有真实用户交互才能触发更新；知识覆盖日常场景后应趋于稳定，而非持续自发探索。
  • 明确把演化限制在上下文层：不修改模型权重、不做架构搜索、不让工具自主进化，仅记录工具使用经验与业务知识，因此错误的最坏后果只是“一条错误参考信息”，而不是模型行为被不可逆改变。

Results

  • 文中声称已在真实数据库上进行了完整演化实验，并提供完整演化日志、Five-Gate 决策记录、质量审计、逐轮知识快照，可比较任意两轮观察知识增长过程。
  • 提供了一个可运行的参考实现 examples/db-investigator/，用于 MySQL 数据库调查，并包含 Python 工具脚本（如 db_query.py、fetch_structure.py、fetch_index.py）；工具层也可替换为 MCP。
  • 给出了明确的上下文控制阈值：_index.md < 40 行；当知识库增长到 5 个主题文件、每个 50–80 行 时，若全量加载会浪费约 250–400 行上下文；因此只路由加载相关文件。单个主题文件超过 ~80 行应拆分，总主题文件超过 8 个应考虑合并。
  • 给出了成熟度目标：当领域知识覆盖 90%+ 的日常场景时，Skill 应收敛到稳定状态，仅在业务变化时再增长。
  • 未提供标准基准数据集上的定量指标，也没有与其他方法在准确率、成功率或成本上的直接数值对比；最强的实证主张是该模式已在真实业务数据库场景中验证，并能通过门控与路由机制实现可控知识演化。
