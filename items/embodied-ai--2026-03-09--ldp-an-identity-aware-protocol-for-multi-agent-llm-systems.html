LDP: An Identity-Aware Protocol for Multi-Agent LLM Systems

LDP提出一种面向多智能体LLM系统的“身份感知”通信协议，把模型身份、质量/成本线索、会话治理、溯源与信任域变成协议层的一等公民。论文给出JamJet插件实现，并用与A2A/随机基线的初步实验说明：该类AI-native协议主要带来效率与治理优势，而非在小规模代理池中稳定提升总质量。

Summary

LDP提出一种面向多智能体LLM系统的“身份感知”通信协议，把模型身份、质量/成本线索、会话治理、溯源与信任域变成协议层的一等公民。论文给出JamJet插件实现，并用与A2A/随机基线的初步实验说明：该类AI-native协议主要带来效率与治理优势，而非在小规模代理池中稳定提升总质量。

Problem

  • 现有多智能体协议如A2A、MCP主要暴露名称、描述、技能等表层信息，却不把模型身份、推理风格、质量校准、成本特性作为协议原语，导致路由和委派决策信息不足。
  • 这很重要，因为多代理LLM系统的效果不仅取决于“找谁做”，还取决于如何通信、如何保留上下文、如何追踪来源、如何施加安全边界；缺失这些会增加延迟、token成本和治理风险。
  • 作者要解决的是：如何设计一个更“AI原生”的代理间协议，使委派更高效、更可治理，并能比现有协议更好地支持复杂多轮协作。

Approach

  • 核心方法是提出LDP (LLM Delegate Protocol)：给每个代理一张更丰富的“身份卡”，显式携带模型家族、质量提示、推理画像、上下文窗口、延迟/成本提示、信任域等信息，让路由器能更合理地挑选代理。
  • 设计渐进式载荷模式：从普通文本（Mode 0）到结构化语义帧（Mode 1），并支持更高级但未充分实证的模式；通信双方先协商最丰富的共同模式，失败时自动回退到更保守模式。
  • 引入governed sessions，即持久化多轮会话：先协商能力、预算、隐私和审计要求，再在会话中连续交换任务，避免每轮都重复发送上下文。
  • 为每个结果附加结构化provenance（谁生成、置信度、是否验证等），并通过trust domains在协议层施加签名、重放保护、策略检查和跨域限制。
  • 实现上，LDP作为JamJet外部插件适配器接入；评测用本地Ollama模型作代理，Gemini 2.5 Flash作LLM judge，对比A2A和随机路由。

Results

  • 路由质量（RQ1）：总体质量上，LDP 6.80±3.62，A2A 7.43±3.54，随机 6.95±3.21（n=30）；LDP vs A2A无显著差异（p=0.56）。作者明确承认：在仅有3个代理的小池中，身份感知路由没有提升总体质量。
  • 延迟收益（RQ1）：在简单任务上，LDP通过把任务路由到轻量模型，实现 2.9s vs 34.8s 的延迟，即约 12×更快；这是论文最明确的实证优势之一。
  • 载荷效率（RQ2）：语义帧（Mode 1）把token从 1215±751 降到 765±510，相对文本减少 37%；相对A2A JSON（1128±804）减少 32%。该token下降显著（p=0.031, d=-0.7）。
  • 载荷延迟/质量（RQ2）：语义帧延迟 14.0s，文本 24.1s，快 42%；质量 5.70±2.93 vs 5.54±4.05，无观察到质量损失（p=0.96）。
  • 溯源（RQ3）：准确provenance与无provenance的决策质量接近（7.65±1.21 vs 7.85±1.84, p=0.47）；但噪声provenance把质量降到 6.85±2.66，且方差更高，说明未经验证的置信度元数据可能比没有元数据更糟。
  • 会话与模拟分析：摘要声称governed sessions在10轮对话下消除 39% token overhead；信任域模拟中攻击检测 96% vs 6%，回退链模拟中任务完成率 100% vs 35%。但作者也强调，这些安全/恢复结果主要反映协议设计属性与模拟分析，并非真实攻击场景实证。
