Summary

这篇论文把LLM上下文窗口重新定义为“L1缓存”而不是完整内存，并提出代理式按需分页系统 Pichay 来管理上下文。核心观点是：当前智能体LLM的大量上下文浪费，本质上是一个未被系统化解决的虚拟内存问题。

Problem

  • 论文要解决的是 LLM/agent 会话中上下文窗口被持续累积的历史、工具定义和旧工具结果占满 的问题，这会带来上下文上限、注意力退化和成本随会话长度恶化。
  • 作者在 857 个生产会话、54,170 次 API 调用、44.5 亿有效输入 token 上测得 21.8% 的 token 属于结构性浪费：未使用工具 schema 11.0%，重复内容 2.2%，陈旧工具结果 8.7%。
  • 这很重要，因为现有做法主要是把上下文窗口做大，但长会话的处理与注意力成本仍会上升，且会话状态无法被系统地保留与回收。

Approach

  • 核心方法是 Pichay：一个放在客户端和推理 API 之间的透明 HTTP 代理，在每轮请求时检查完整消息流，自动把“暂时不用但可再取回”的内容从上下文中移除，并保留一个简短的“分页标记”。
  • 当模型之后再次请求同一内容（例如再次读取被移除的文件）时，系统把这视为 page fault（缺页）；也就是模型在告诉系统“我还需要刚才被移走的内容”。
  • 系统区分两类内容：garbage collection 用于不能稳定重取的临时工具输出；paging 用于像文件读取这类可按地址重取的内容。这样才能正确测量真正的缺页率。
  • 它使用非常简单的 FIFO 按用户轮次年龄淘汰（实验中 τ=4 turns，s_min=500 bytes），并加入 fault-driven pinning：某页一旦因淘汰而产生过缺页，之后同版本内容就被固定，不再重复淘汰。
  • 进一步地，论文还提出 协作式内存管理：通过 phantom tools 和 cleanup tags，让模型主动释放冷内容、请求恢复被分页内容，或把历史对话压缩为摘要，形成 L1/L2/L3/L4 的完整内存层级设计。

Results

  • 在生产语料测量中，作者报告 21.8% 结构性浪费，细分为：未使用工具 schema 11.0%、重复内容 2.2%、陈旧工具结果 8.7%；其中陈旧工具结果被重复处理的中位放大倍数达到 84.4×。
  • 离线重放实验覆盖 140 万次模拟淘汰，报告 缺页率仅 0.0254%，说明大量被移除内容实际上不会被再次需要。
  • 在线生产部署中，在 681-turn 会话里，系统将上下文消耗 最多降低 93%，从 5,038KB 降到 339KB。
  • 论文还给出一个具体生产案例：会话剩余可用上下文从 7% 提升到 43%，把“即将因上下文耗尽而死亡”的会话延长为仍有较大余量的会话。
  • 在极端持续压力下，系统仍能运行，但会出现经典 thrashing（抖动）：被淘汰内容反复 fault-in，说明当工作集超过常驻集时，该方法也遵循操作系统已知规律。
  • L3 会话压缩与 L4 跨会话持久记忆已被设计/部分实现，但文中说明 只有前两级（L1 淘汰、L2 pinning）完成了定量评估；L3 尚未大规模评测。
