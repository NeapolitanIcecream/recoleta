Is Code Still Relevant?

这篇文章不是学术论文，而是一篇基于一线开发经验的行业观察：作者认为 AI 代码代理已经显著改变软件开发流程，使“手写代码”从核心活动退化为被监督的执行环节。核心判断是代码仍然重要，但程序员的价值将更多转向规格定义、架构判断、测试与调试。

Summary

这篇文章不是学术论文，而是一篇基于一线开发经验的行业观察：作者认为 AI 代码代理已经显著改变软件开发流程，使“手写代码”从核心活动退化为被监督的执行环节。核心判断是代码仍然重要，但程序员的价值将更多转向规格定义、架构判断、测试与调试。

Problem

  • 文章讨论的问题是：当 AI 能自动生成和修改大量代码时，代码本身是否仍然是软件工程的核心资产，以及程序员还是否重要。
  • 这之所以重要，是因为它直接影响软件工程的工作流、技能要求、招聘结构和初级开发者的成长路径。
  • 作者还指出一个现实风险：如果操作者不理解实现细节，AI 更容易引入回归、临时修补和系统复杂度上升的问题。

Approach

  • 核心方法不是提出新算法，而是基于作者 16 年编程经验 与过去 6 个月 使用 Copilot、ChatGPT、Cursor、Claude Code 的连续实践，比较 AI 前后开发模式的变化。
  • 机制上，作者描述了一种新的最简单工作流：人负责提出需求和约束，AI 负责写代码，人再做审阅、测试和纠偏。
  • 作者把开发活动重新分层：低层代码编写越来越自动化，而高层任务如系统结构、文件划分、架构决策、手工测试、规格表达和代码评审仍由人主导。
  • 作者特别强调，AI 代理能自动读取项目上下文并执行改动，这比早期补全式工具更强，但仍常在复杂设计问题上“钻牛角尖”，需要有经验的开发者显式引导其退后重构。

Results

  • 最强的量化信号是作者的个人实践结论：过去 6 个月 里，作者称自己没有完全手写过一行代码，并且在使用 Claude Code 后“几乎停止手动写代码”。
  • 作者报告了明确的效率主张：手动写代码现在感觉“太慢”，说明在其日常开发中，AI 代理已替代大部分编码劳动；但文中没有提供标准基准、数据集或正式对照实验。
  • 工作流层面的具体结果是：开发时间的重心已转向手工测试、设计决策、写规格/提示词，而不是逐行实现代码。
  • 可靠性层面的结果是：AI 通常能快速找 bug 并从零实现功能，但也会出现错误方向探索、循环依赖的临时修补、以及潜在回归，因此作者仍坚持逐步审阅代码而非完全 YOLO 模式。
  • 对“程序员是否会消失”的结论性主张是：程序员不会消失，但角色会迁移到系统思维、架构权衡、调试复杂问题、把模糊需求转成可执行规格这些更高层能力上。
  • 对入门者的最具体判断是：过去 10 年 里只深挖一个流行工具也许足以找工作，但作者认为未来这种路径大概率不再够用，需要更宽的能力结构。
