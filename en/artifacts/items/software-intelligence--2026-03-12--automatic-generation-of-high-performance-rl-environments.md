# Automatic Generation of High-Performance RL Environments

## Summary
This paper proposes a general workflow for automatically translating reinforcement learning environments into high-performance implementations using coding agents, while ensuring semantic equivalence through hierarchical verification. The core claim is: high-performance RL environments that previously required months of manual engineering can now be generated automatically for under $10 in agent compute cost.

## Problem
- In RL training, environment simulation often accounts for **50%–90%** of wall-clock time; complex environments (such as Pokemon Showdown and Game Boy emulators) can severely bottleneck training.
- Existing high-performance environments usually depend on **manual rewrites tailored to a single domain**, making them costly and time-consuming, and difficult to turn into a standard workflow.
- If automatic translation is unreliable, even small semantic deviations can corrupt the training signal, so the goal is not only speed, but also proving **behavioral consistency with the original environment**.

## Approach
- A **general translation recipe** is used to drive coding agents: the reference environment is decomposed into modules and translated module by module into **JAX** (suited to GPU-parallel pure functions) or **Rust** (suited to stateful, memory-intensive CPU parallelism).
- A four-level **hierarchical verification** loop is used: L1 property tests, L2 interaction tests, L3 full rollout comparisons, and L4 cross-backend policy transfer; failure at any level triggers targeted repair.
- Put simply, the method is essentially: **first let the agent write the code, then use increasingly strict tests to iteratively force the implementation toward something “the same as the original environment, but faster.”**
- Humans are mainly responsible for prompts, module decomposition, and test structure; the target code is claimed to be **entirely generated by the agent**, with human intervention only after up to 50 rounds without progress.

## Results
- Validated on **5 environments**, covering discrete games, continuous control, hardware emulation, and multi-agent systems; translation costs were approximately **$0.05–$6**, all below **$10**.
- **PokeJAX** (TypeScript→JAX, 100K+ LoC): random-action throughput **21K → 500M SPS**, a **23,810×** improvement; PPO training **681 → 15.2M SPS**, a **22,320×** improvement; claimed to be the first **GPU-parallel Pokemon battle simulator**.
- **Pong** (optimized C baseline→Rust/JAX): GRU rollout **4.5M → 140M SPS (31×)**; GRU PPO **854K → 35.5M SPS (42×)**, showing large end-to-end gains even for already optimized environments.
- **HalfCheetah**: versus Gymnasium, **45K → 1.66M SPS (37×)**; versus Brax at matched batch size, **5.0×**; versus Google MJX, reaches **1.04× throughput parity** (**1.66M vs 1.6M SPS**).
- **TCGJax** (a new environment built from web rules): random-action **140K → 717K SPS (5.1×)**; PPO **23K → 153K SPS (6.6×)**; the authors claim this is the first deployable **JAX Pokemon TCG engine**.
- On semantic consistency: all **5/5** environments passed **100 episodes** of L3 rollout comparison and also passed L4 cross-backend policy equivalence checks; for example, PokeJAX shows **exactly identical** cross-backend win rates, and HalfCheetah is judged equivalent using **TOST p<0.05**. Additional training-time profiling shows that for a **200M-parameter** model, environment overhead drops to **≤4%** of training time.
