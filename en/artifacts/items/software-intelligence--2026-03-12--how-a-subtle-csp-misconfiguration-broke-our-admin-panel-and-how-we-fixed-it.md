# How a subtle CSP misconfiguration broke our admin panel and how we fixed it

## Summary
This is an engineering retrospective on a CSP configuration mistake in Rails + ActiveAdmin that caused admin forms to fail silently. The article explains how they fixed inline scripts blocked by a strict `script-src 'self'` policy using a nonce, without weakening the security policy.

## Problem
- The problem to solve was that a strict CSP configuration blocked the inline JavaScript relied on by ActiveAdmin, causing key admin workflows such as dynamic forms, validation, and button states to fail, while the interface still appeared normal on the surface.
- This matters because it affects both **security** and **usability**: CSP is intended to prevent XSS, but an incomplete configuration can silently disable legitimate functionality and disrupt business operations.
- The issue also showed that routine QA and automated checks can easily miss frontend failures where the visuals look normal but behavior is broken.

## Approach
- The core method was simple: generate a random CSP nonce for each request, include it both in the `script-src` directive of the response header and in inline `<script>` tags on the page, so the browser executes only trusted scripts with the correct nonce.
- The article first investigated the browser console and response headers, confirming that the current policy was only `script-src 'self'`, with no nonce, hash, or exceptions, so all inline scripts were being blocked by the browser.
- It then compared four options: externalizing all JS, adding `unsafe-inline`, adding a hash for each inline script, and using a nonce; nonce was ultimately chosen because it offered the best balance of security, maintainability, and implementation cost.
- The specific implementation included enabling `content_security_policy_nonce_generator` in the Rails CSP initializer, exposing the nonce to ActiveAdmin views, and explicitly adding a `nonce` attribute to inline scripts generated by `script do ... end`.
- In addition to the fix, they also added end-to-end tests for CSP scenarios, a QA checklist, and internal implementation guidelines to catch this kind of silent failure earlier.

## Results
- The article does not provide standardized benchmark experiments, public datasets, or performance metrics, so there are **no quantitative results** to report.
- The clear engineering result was that after the fix, the response headers included `Content-Security-Policy: script-src 'self' 'nonce-...'`, and the inline scripts carried a matching `nonce="..."` attribute.
- After the fix, ActiveAdmin’s dynamic form behavior was “fully restored”: field show/hide behavior, form validation, and button enablement logic returned to normal; before the fix, these functions had stopped working in production.
- Compared with the `unsafe-inline` approach, the final solution did not relax CSP rules; compared with the hash approach, it avoided the fragility of scripts breaking whenever they changed; compared with fully externalizing JS, it required fewer changes.
- An additional outcome was process improvement: the team added lightweight E2E checks and CSP-related QA validation to reduce similar regressions in the future.
