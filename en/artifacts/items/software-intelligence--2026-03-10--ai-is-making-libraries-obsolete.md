# AI Is Making Libraries Obsolete

## Summary
This is an opinion piece about how “AI is making general-purpose software libraries gradually unnecessary,” rather than a strict academic paper. The author argues that as agentic development and code generation capabilities improve, many practices that once relied on general-purpose libraries, frameworks, and protocols to encapsulate complexity will be replaced by on-demand generated custom implementations.

## Problem
- The question the author is trying to answer is: when AI can directly generate SQL, CSS, workflows, and glue code, **are general abstraction layers such as ORMs, CSS frameworks, static site tools, and even MCP losing their value**.
- This matters because much of the complexity in the modern software stack comes from the “abstraction cost paid for generality”: learning cost, performance loss, debugging burden, dependency maintenance, and framework lock-in.
- The article also points to a deeper issue: if more and more code is generated by AI, then **responsibility, maintainability, and human review** will become more important software engineering questions than “whether to reuse a library.”

## Approach
- The core method is not to propose a new algorithm, but to offer a mechanistic judgment based on the author's development practice: **when AI can instantly generate implementations that are good enough for the need, the value of prepackaged general-purpose libraries declines**.
- At the simplest level, the mechanism is this: previously developers used libraries to avoid handwriting complex code; now developers can have agents directly generate code that “fits the current need exactly,” so the intermediate abstraction layer becomes an extra burden.
- The author supports this with multiple examples: ORMs can be replaced by parameterized SQL generated by LLMs; CSS frameworks can be replaced by native CSS generated by agents; MCP wrapper layers can be replaced by agent workflows that “read CLI documentation and then call commands directly.”
- The article further proposes that in the future, reusable core assets may shift from “code packages” to “documentation, blog posts, design rationale, and decision processes,” because this content is more helpful for agents to understand intent and generate implementations.
- At the same time, the author emphasizes that AI should be a tool rather than the bearer of responsibility: humans still need to take responsibility for, review, and maintain generated software.

## Results
- **No formal experiments, datasets, or reproducible empirical results are provided**; the entire text is basically a position piece based on experiential observations and case narratives.
- On SQL/ORM, the author's strong claim is that LLMs can now generate “**parameterized, injection-safe**” SQL, thereby weakening the necessity of ORMs; however, the article **does not provide numbers for accuracy, defect rate, performance, or comparative baselines**.
- On the frontend side, the author claims that the **CSS for the current blog was generated by an agent**, presenting it as an example that UI can be built without Bootstrap/Bulma/Fluent UI; however, **no comparison data is given for bundle size, performance, or development time**.
- On tooling protocols, the author believes **MCP is already showing its age**, because agents can directly read man pages and invoke CLIs such as `gh`, `az`, and `kubectl`; again, there is **no quantitative comparison of success rate, latency, or maintenance cost between MCP and direct CLI invocation**.
- On knowledge representation, the specific observation the author gives is that a Microsoft friend had an AI agent use his **ActivityPub blog series** rather than a GitHub repository or the W3C spec to implement functionality; from this, the author concludes that “**prose with context and decision rationale is more effective for agents**”; however, **no experimental sample size or success-rate figures are provided**.
- The strongest overall conclusion is that the software stack is shifting from “shared general-purpose code” to “shared knowledge and context,” and that the truly scarce differentiating capability will become **human responsibility and review**, not more abstraction libraries themselves.
