# AI Is Making Libraries Obsolete

## Summary
This is an opinion piece about how AI is changing the software development toolchain, rather than an academic paper. The author argues that as LLMs and agent-based development advance, the value of many general-purpose software libraries and frameworks will be weakened by custom code generated on demand.

## Problem
- The author believes that general-purpose libraries such as ORMs, CSS frameworks, and static site generators are often justified in the name of “abstracting complexity,” yet in complex or high-performance scenarios they introduce additional maintenance costs, performance overhead, and framework lock-in.
- Traditional “one-size-fits-all” libraries are often “usable by everyone but not a perfect fit for anyone.” Developers must understand both the business logic and the library itself, making debugging, optimization, and customization more difficult.
- As AI-generated code becomes more capable, the question becomes: if models can directly generate the required SQL, CSS, or workflow scripts, why still depend on heavy intermediate abstraction layers? This has implications for how the future software stack, development efficiency, and responsibility attribution will be restructured.

## Approach
- The core mechanism is simple: use LLMs/agents to directly generate code or commands **for specific tasks**, rather than first learning and adapting to the abstract interfaces of general-purpose libraries.
- The author supports this with several examples: having AI write parameterized SQL directly to replace the role of ORMs in complex queries; having AI generate CSS directly to replace UI frameworks such as Bootstrap; and directly invoking existing CLI documentation and commands instead of wrapping every tool in an additional MCP protocol layer.
- The article further suggests that in the future, the more important asset may not be reusable code libraries, but written knowledge that helps AI understand design intent, such as blogs, process descriptions, and decision rationales.
- At the same time, the author emphasizes that humans must still remain responsible for AI-generated software, because security, defects, and operational issues cannot be excused with “the AI wrote it.”

## Results
- The article **does not provide formal experiments, benchmark data, or reproducible quantitative results**, so it should not be treated as a validated research conclusion.
- One of the strongest concrete claims is that the author says Claude can now directly generate “parameterized, injection-safe” SQL, thereby reducing the burden of ORM configuration, mapping, and tuning; however, **no accuracy, performance, or comparative figures are provided**.
- Another concrete example is that the author says the CSS for their blog was generated by AI, and that they rewrote the entire publishing workflow, replacing Hugo with a simplified .NET solution; however, **no figures are reported for build time, code size, failure rate, or maintenance cost**.
- The judgment about MCP is also a qualitative conclusion: the author believes agents can directly read CLI man pages and use tools such as `gh`, `az`, and `kubectl`, so the value of the MCP wrapper layer declines; **no comparative data is provided on protocol efficiency or task success rates**.
- As for the claim that “blogs are better than code repositories for teaching AI,” the basis is that a friend once pointed an AI agent to their ActivityPub blog series for implementation; this is **anecdotal evidence** and lacks systematic evaluation.
