# Intent Expression and Agent Skills

## Summary
This article proposes a document generation and review skill pipeline for agentic coding, using structured documents to turn “intent in the human mind” into input that an agent can execute reliably. The core argument is: when an agent writes the code, clearly expressing intent and reviewing in stages improves development leverage more than simply asking the model to “help write.”

## Problem
- The problem it aims to solve is that when developers plug AI agents into the traditional workflow of “first think it through in your head, then write code directly,” they often do not gain efficiency, because the key bottleneck becomes **how to externalize intent clearly for the agent**.
- This matters because when code is primarily generated by an agent, documentation is no longer just an after-the-fact artifact, but the core interface that determines whether the agent can correctly implement, reuse, and continuously iterate on requirements.
- The article also points out a practical risk: LLMs can create the illusion that “the document is already complete,” and may introduce hallucinations, fabricated links, incorrect assumptions, or redundant filler. Without checkpoints, these problems can be amplified at the implementation stage.

## Approach
- The core method is to break document work into 3 reusable skills: `flesh-out` (expand raw notes into a structured document), `review-steps` (polish in stages, validate, compare against best practices, and verify links), and `strong-edit` (criticize first, then revise, strengthening the core argument and readability).
- Put simply: first have the agent **understand what you mean**, then have it **organize and check the document**, and finally have it **challenge the weakest parts of the document**; pause at each stage and wait for human confirmation before continuing, to avoid errors propagating all the way through.
- This mechanism depends on “docs as code”: documentation is versioned together with source code, making it a shared, continuously updated reference for both humans and agents rather than a one-off conversation.
- The design emphasizes “clear division of labor”: humans provide new ideas, engineering judgment, and final decisions; the agent handles structuring, language polishing, search framing, consistency checks, and identifying obvious gaps.
- The article also summarizes several behavioral patterns: skill-file scaffolding is more stable than putting rules into `CLAUDE.md/AGENTS.md`; different terms such as “review,” “flesh-out,” and “strong-edit” significantly shape model behavior.

## Results
- In the writing case discussed in the article, `/flesh-out` expanded **727 words** of raw notes to **2279 words**, about **3.1x**; subsequent human editing reduced the word count from **2279** to **2007**, a cut of about **12%**, mainly by removing agent-generated filler and hedging.
- In the same case, `/review-steps` expanded the content again to **2257 words**, adding external references and observations; `/strong-edit` ultimately compressed it to about **1930 words**, while improving the opening argument, sentence force, and overall reasoning structure.
- One key negative result led to a process improvement: `/review-steps` once added a **fabricated URL**, and that error was not caught through either human editing or `strong-edit`, ultimately prompting the author to add **Stage 6: verify links**, requiring links to be checked one by one to ensure they resolve and support the claims being made.
- In another CPU model design case, `/flesh-out` expanded **169 words** of raw notes to **1962 words**, about **11.6x**; the agent then researched the source code and found **21+** user-space assembly files spread across **10** library modules and **3** applications, overturning the earlier core assumption that “only the kernel directly uses the ISA.”
- The author gives a rough rule of thumb: in Stage 0 of `flesh-out`’s “Extract Idea” and `strong-edit`’s “Core Argument,” the agent’s grasp of the core intent is about **70%** correct, with the rest clarified through questions.
- The article also cites a developer trust statistic: developers’ trust in AI accuracy is only **43%**; based on this, the author argues that staged checkpoints and traceability of information sources are key to reducing the risk of blind trust.
