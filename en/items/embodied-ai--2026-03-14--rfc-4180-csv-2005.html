Summary

This RFC provides a common definition for the CSV format, which had long been widely used but lacked a unified specification, and formally registers the text/csv MIME type. Its main value is improving interoperability when exchanging tabular data between different programs, rather than proposing new algorithms or models.

Problem

  • CSV was widely used for spreadsheets and data conversion, but there had previously been no formal, unified specification, leading different implementations to interpret format details differently.
  • IANA had previously not registered a standard MIME type for CSV, and different systems used different media types, affecting data exchange and protocol-level compatibility.
  • This matters because CSV is a common format for sharing structured text data across software and platforms; the lack of a common convention can directly cause parsing errors and interoperability problems.

Approach

  • The document summarizes the CSV conventions that “most implementations commonly follow” and writes them down as a set of explicit format rules: records line by line, fields separated by commas, optional headers, consistent field counts, and so on.
  • It provides the most important unified explanation of quoting and escaping mechanisms: fields containing commas, line breaks, or double quotes should be enclosed in double quotes; double quotes inside a field are escaped using two double quotes.
  • It uses ABNF syntax to formally describe the CSV file structure, allowing implementers to write parsers or validators accordingly.
  • It formally registers the text/csv MIME type and defines the optional parameters charset and header, thereby providing a unified identifier for network transmission and system integration.

Results

  • The document’s main outcome is a standardized definition rather than experimental evaluation; it does not provide quantitative experimental results, benchmark datasets, or performance figures.
  • It clearly states 7 core format rules, covering key details such as record separation, optional headers, consistency in field counts, double-quote enclosure, and escaping.
  • It provides a complete ABNF grammar: for example, file = [header CRLF] record *(CRLF record) [CRLF], turning previously vague conventions into a machine-implementable syntactic description.
  • It formally completes the registration of the text/csv MIME type and specifies 2 optional parameters: charset and header (with header taking the values present or absent).
  • It gives a clear requirement for newline conventions: the media type uses CRLF to represent line endings according to MIME text rules, while also noting that other newline values may exist in real implementations.
  • Its strongest concrete claim is that, by providing a unified format description and MIME registration, it can reduce interoperability differences among CSV implementations caused by the absence of a single specification.
