Summary

This is a tool that directly translates a specific subset of WebAssembly modules into a single Go source file, aiming to preserve Wasm semantics and achieve practical performance while depending only on the Go standard library. It is more like an engineering-oriented compiler/transpiler than a general-purpose Wasm runtime.

Problem

  • The problem it solves is: how to turn a Wasm module directly into native Go code that can be embedded in a Go project, without relying on an extra runtime or third-party libraries.
  • This matters because many software systems want to integrate existing Wasm artifacts (for example, modules generated by clang) into Go applications while preserving behaviors such as exported functions, memory, and global variables.
  • The existing difficulty is that Wasm and Go differ significantly in control flow, type systems, floating-point semantics, and memory models, making direct and semantically correct translation nontrivial.

Approach

  • The input is a Wasm module, and the output is a single self-contained Go file; this file exports a Module struct and a New initialization function, with Wasm exported functions mapped to Module methods and imports mapped to interfaces accepted by New.
  • The core method is to convert Wasm’s stack-machine execution model into a Go-friendlier stack-to-register form, then use goto and labels to implement Wasm control flow, thereby preserving semantics as much as possible.
  • The tool supports only a practical subset aimed at clang-produced artifacts: it covers most of Wasm 1.0 plus some Wasm 2.0 features, such as bulk memory, reference types, nontrapping float-to-int, sign-extension, and multi-values.
  • To ensure semantic correctness, the generated code explicitly handles type differences between Go and Wasm, floating-point literal representation, an optional NaN canonicalization setting, and endianness targets; on little-endian CPUs it can also use unsafe to generate faster code while still preserving bounds checks.

Results

  • The concrete engineering result is: a Wasm module can be translated into a single Go source file via wasm2go < input.wasm > output.go, with no dependencies beyond the standard library.
  • In terms of support scope, the author states that the current target is a useful Wasm subset produced by clang: it includes most Wasm 1.0 features and supports 5 categories of Wasm 2.0 subset features (bulk memory operations, reference types, nontrapping float-to-int conversions, sign-extension operators, multi-values).
  • In terms of constraints, unsupported examples include 3 categories: export aliasing, export conflicts after simple name mangling, and importing tables or globals.
  • On performance, the text provides only a qualitative conclusion: using unsafe on little-endian CPUs can generate “much faster code,” but the excerpt does not provide specific benchmark numbers, datasets, or quantitative comparisons against a baseline.
  • On correctness, the author emphasizes that results should be evaluated based on the assembly generated by the Go compiler rather than source readability, and claims that the generated code follows unsafe rules and that all memory accesses are bounds-checked.
