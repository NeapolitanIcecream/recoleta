Summary

This paper systematically analyzes why LLMs fail at automated security patch generation, finding that most models do not fail because they cannot write code, but because they do not understand how to truly fix vulnerabilities. The authors also propose a continuous scoring metric, SRS, to measure the quality of “partial success” in security repair.

Problem

  • The problem the paper addresses is: why exactly do LLM-generated security patches fail, and why is traditional evaluation based only on whether tests pass insufficient.
  • This matters because even if a patch passes conventional functional tests, it may still remain exploitable by attackers, and thus be mistakenly judged as “deployable” in CI/CD.
  • Security repair is different from ordinary program repair: it must simultaneously verify that the patch compiles, blocks attacks, and does not break original functionality.

Approach

  • The authors evaluate 64 reproducible Java vulnerabilities on the Vul4J benchmark, collecting a total of 319 patches generated by Gemini 3.0 Flash using zero-shot prompts.
  • They use a “three-axis evaluation”: whether compilation succeeds, whether security passes PoV attack tests and reduces Semgrep alerts, and whether functionality passes the developer test suite.
  • Patches are divided into 5 categories: Correct & Secure, Compilation Error, Security Failure, Functionality Failure, Insecure & Breaking, for failure mode analysis.
  • They propose the Security Repair Score (SRS): first compute a security score and a functionality score separately, then combine them into a weighted continuous metric from 0 to 1 when the patch compiles, to capture cases that are “partially fixed but not fully correct.”
  • They further analyze which factors affect repair difficulty, including CWE type, code complexity, and human patch size.

Results

  • Among the 319 patches, only 79 (24.8%) are fully correct and secure; 164 (51.4%) fail in both security and functionality; 42 (13.2%) fail to compile; 33 (10.3%) preserve functionality but remain insecure; and only 1 (0.3%) is secure but breaks functionality.
  • The paper argues that the dominant failure mode is semantic misunderstanding: models can usually generate syntactically correct Java code (overall compilation rate 86.8%), but they adopt the wrong repair strategy, rather than simply making syntax errors.
  • The continuous metric shows that models are better at preserving functionality than fixing security: average Functionality Score = 0.832, average Security Score = 0.251, average SRS = 0.542; the functionality score is about 3.3× the security score.
  • The success distribution is clearly bimodal: Perfect (SRS=1.0) accounts for 79 (24.8%), Near-success (0.8≤SRS<1.0) includes only 1 (0.3%), Partial accounts for 188 (58.9%), and Complete failure (SRS=0) accounts for 51 (16.0%). Based on this, the authors argue that security patch generation is more like “complete success or complete failure,” with almost no near-success cases.
  • Vulnerability type strongly affects difficulty: among CWE categories with larger sample sizes, CWE-20 Input Validation has a fix rate of 0% (compilation rate 95%, SRS 0.469), while CWE-835 Infinite Loop has a fix rate of 45% (compilation rate 100%, SRS 0.725); CWE-611 XXE has a fix rate of 40%, and CWE-264 Permissions has a fix rate of 15% with a high proportion of cases that are “functional but insecure.”
  • For difficulty prediction, human patch size shows a significant negative correlation with average SRS (Spearman ρ = -0.331, p = 0.008), while LOC and cyclomatic complexity show no significant correlation with success rate, indicating that the main challenge lies in “understanding what to change,” not in structural code complexity.
