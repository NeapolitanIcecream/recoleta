Summary

This paper proposes a core formal framework for C software product lines: LC is used to characterize a subset of C without a preprocessor, while CLC extends it with #define/#if-style variation annotations and a corresponding family-based type system. Its goal is to statically guarantee, without enumerating all variants, that all product variants generated by the preprocessor are well-typed C programs.

Problem

  • C software product lines often implement variation through preprocessor directives, but the number of feature combinations can grow exponentially, making it impractical in practice to generate and check every variant individually.
  • Although existing tools can analyze C code with preprocessors, the authors argue that there is a lack of a formal family-based type-checking framework for this problem built on a core calculus that is truly a subset of C.
  • A simple theoretical foundation is needed that can both express key C mechanisms such as structs, pointers, and heap allocation, and prove that all valid products are type-safe; this is important for both analysis and teaching.

Approach

  • Proposes Lightweight C (LC): a formal core subset of ANSI C covering mechanisms such as structs, pointers, conditional expressions, expression sequences, struct field assignment, and dynamic memory allocation/deallocation.
  • Proposes Colored LC (CLC): LC extended with feature-condition annotations, abstracting the #define, #if/#ifdef/#ifndef-style variation selection of the C preprocessor.
  • Uses family-based type checking: instead of type-checking each product separately, it checks the entire product line at once over the shared code base in combination with the feature model and annotation information.
  • The proof objective is: if a CLC software product line type-checks, then all LC variants generated by the preprocessor also type-check, thereby establishing the guarantee that “well-typed product line ⇒ all variants well-typed.”
  • Uses a queue example with 3 features (LAST, OCCURS, INTERVAL) and 6 products to illustrate how variation annotations can be applied to struct definitions, functions, parameters, and expressions in C code.

Results

  • The paper’s core result is a formal guarantee: the authors claim in Section 6 to prove that all variants generated by a well-typed CLC SPL are well-typed LC programs.
  • The provided running example includes 3 features and 6 products/variants; the authors state that these 6 example variants are all well-typed LC programs.
  • The paper explicitly contrasts this with the product explosion problem: for example, 64 features correspond to at most 2^64 products, and Gentoo statistics mention 671617 features distributed over 36197 feature models, corresponding to at most 2^671617 products; this is used to argue that per-variant checking is infeasible and that family-based checking is necessary.
  • The excerpt does not provide experimental evaluation, runtime, accuracy, or quantitative comparison results with existing tools (such as TypeChef) on benchmark datasets.
  • The strongest concrete claim is that, to the authors’ knowledge, this is the first formalization of family-based type checking for annotative C software product lines built on a core language that is a “true subset” of C, and that LC/CLC can also serve as a foundation for subsequent research on analyses such as heap safety.
