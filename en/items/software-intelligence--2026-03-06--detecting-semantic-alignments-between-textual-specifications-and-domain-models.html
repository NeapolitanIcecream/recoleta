Summary

This paper proposes a method for detecting semantic alignment/misalignment between textual requirements specifications and domain model elements, with the goal of helping modelers verify whether a model is correct and providing supporting evidence sentences. The core idea is to first convert model elements into natural language, and then use an LLM to judge whether they are equivalent to, contradictory to, or subsumed by requirement sentences.

Problem

  • Domain models are important in the early stages of software engineering, but constructing correct models from natural-language requirements and establishing clear traceability links is especially difficult for novices.
  • Models generated by existing automatic/semi-automatic modeling methods still require manual validation; moreover, there is often no single uniquely correct model for the same requirement, which makes validation harder.
  • A method is needed that can indicate which model elements are correct, which are incorrect, and which requirement sentences provide the evidence, to support teaching, modeling assistance, and offline quality checking.

Approach

  • The method first uses rule-based NLP to preprocess textual requirements, extracting textual concepts, relations, and their corresponding sentences; it then slices the domain model to extract minimal context for each attribute, association, inheritance relation, enumeration, etc.
  • For each model element, the system uses rules to generate a simple natural-language description from the model slice, for example turning an attribute/association into a sentence like “a car has a plate.”
  • Then, through a semantic matcher, the model element is aligned with potentially relevant sentences in the requirements, producing a set of candidate evidence sentences for each element.
  • Finally, GPT-4 zero-shot is used to make three types of judgments: semantic equivalence, semantic contradiction, and semantic inclusion; each type uses multiple semantic-equivalence prompt variants, and relative majority voting is used to reduce prompt sensitivity.
  • The classification rules are straightforward: if any sentence is equivalent to the model description, it is classified as aligned; if there is no equivalence but there is contradiction, it is classified as misaligned; if there is no contradiction but the requirement sentence semantically includes the model sentence, it is also classified as aligned; otherwise it is unclassified.

Results

  • The paper claims that, in evaluations on multiple examples from the literature across different domains, as well as on erroneous models systematically generated from correct models through mutation, it can identify alignments and misalignments with precision close to 1.0.
  • Recall is about 78%, meaning the system can make judgments for more than 3/4 of model elements, though some will still remain unclassified.
  • Runtime is about 18 seconds to 1 minute per model element, indicating that the method is better suited for modeling assistance or offline validation rather than extremely low-latency scenarios.
  • The paper also states that its advantage is that it almost never misclassifies elements, and that it returns the original requirement sentences supporting the decision, making it convenient to provide positive feedback or warnings in modeling tools.
  • The abstract does not provide more fine-grained benchmark comparison figures (such as direct comparisons with other methods on the same dataset), so the strongest quantitative conclusions are mainly precision≈1, recall≈78%, 18s–1min/element.
