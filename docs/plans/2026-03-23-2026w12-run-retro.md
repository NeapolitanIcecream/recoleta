## Findings

### 1. Billing under-reported relative to provider-side spend

#### Symptom

- Repo-side measurable spend was only `2.33458234 USD`.
- Provider-side dashboard showed roughly `30 USD`.
- All seven explicit `analyze` runs reported `cost_missing`, even though token
  counts were present.

#### Root Cause

There were two independent causes.

1. The direct LiteLLM completion paths only checked `_hidden_params.response_cost`
   and ignored provider-reported usage cost fields.
   - In the failing `analyze` / `translate` / doctor paths, Recoleta first looked
     for `_hidden_params.response_cost`.
   - In the observed gateway responses, that field was `None`, but `usage.cost`
     and `usage.cost_details.upstream_inference_cost` were present.
   - Because those usage fields were ignored, calls with real provider-side spend
     still landed in `cost_missing`.

2. The fallback pricing path depended on the resolved response model instead of
   the configured pricing model.
   - Our configured model is `openai/openai/gpt-5.4`.
   - The actual response model returned by the gateway was
     `openai/gpt-5.4-20260305`.
   - LiteLLM can price `openai/gpt-5.4` / `gpt-5.4`, but not the dated resolved
     form, so `completion_cost(completion_response=response)` failed.
   - `recoleta/rag/agent.py` and `recoleta/rag/ideas_agent.py` avoided this by
     estimating from token counts against the configured model, so trends / ideas
     billing succeeded while analyze billing failed.

This explains the local mismatch:

- repo-native measurable lower bound:
  - `trends_llm`: `1.9373075 USD`
  - `ideas_llm`: `0.390385 USD`
  - `triage_embeddings`: `0.00405006 USD`
  - `trends_embeddings`: `0.00283978 USD`
- missing local analyze spend:
  - explicit daily analyze runs: `952` calls, `4,038,715` input tokens, `572,904`
    output tokens
  - hidden analyze inside the `trends-week` run: `523` calls, `2,069,117` input
    tokens, `308,859` output tokens

Using the local LiteLLM `gpt-5.4` price table as a best-effort estimate:

- explicit analyze spend was about `18.6903475 USD`
- hidden analyze inside `trends-week` was about `9.8056775 USD`
- full executed workflow was therefore about `30.83060734 USD`

That estimate aligns with the upstream provider-side observation.

#### Fixes

Implemented:

- Added a shared LLM billing helper in `recoleta/llm_costs.py`.
- The helper now resolves cost in this order:
  - `_hidden_params.response_cost`
  - `usage.cost`
  - `usage.cost_details.upstream_inference_cost`
  - token-based estimation against normalized configured-model candidates
- Wired that helper into:
  - `recoleta/analyzer.py`
  - `recoleta/translation.py`
  - `recoleta/cli/maintenance.py`
- Reused the same token-based pricing helper from:
  - `recoleta/rag/agent.py`
  - `recoleta/rag/ideas_agent.py`

Follow-up still worth considering:

- Add `billing_model_alias` or `pricing_model_alias` if we expect more gateways
  whose request model and pricing model intentionally differ.
  - Example: map `openai/openai/gpt-5.4` to `openai/gpt-5.4`.

Recommended reporting improvement:

- Show two values in billing output:
  - `measured_cost_usd`
  - `estimated_missing_cost_usd` when token counts are present but price lookup failed
- That makes “partial billing” explicit and avoids a false sense of cheapness.

#### Verification

- Added regression tests covering:
  - `usage.cost` recovery for analyze
  - token-based fallback when the response model is a dated resolved variant
  - reuse of the same logic in translation and doctor ping
- Ran a live minimal request against the configured gateway and confirmed:
  - configured model: `openai/openai/gpt-5.4`
  - resolved model: `openai/gpt-5.4-20260305`
  - `_hidden_params.response_cost`: `None`
  - `usage.cost`: `0.0002175`
  - measured-cost recovery returns `0.0002175`
  - token-based fallback with `usage.cost` removed also returns `0.0002175`

### 2. `trends-week` is much slower than its command surface suggests

#### Symptom

- `trends-week` took `37m20s`.
- For long stretches there was no user-visible progress.
- The run looked suspiciously “stuck”, but it was still performing work.

#### Root Cause

The main issue is not a hang. It is hidden repeated work.

1. `run_trends_stage()` always prepares backlog before trend synthesis unless
   `reuse_existing_corpus=True`.
   - In `recoleta/pipeline/trends_stage.py`, `_prepare_period_backlog()` always
     calls:
     - `service.prepare(...)`
     - `service.analyze(...)`
   - This happens for `day`, `week`, and `month`, not only for `day`.

2. Weekly backfill recursively re-enters `run_trends_stage()` for every day in the
   week.
   - In the `week` branch, backfill loops across seven days.
   - Each day backfill call re-runs the same `prepare + analyze + index + trend`
     flow unless `reuse_existing_corpus=True`.

3. Topic-stream mode multiplies the whole sequence by stream count.
   - `run_trends_topic_streams_stage()` loops over every configured stream and
     calls `run_trends_stage()` separately.
   - With two streams, the hidden upstream work is effectively duplicated.

4. The `trends-week` run confirms this behavior from metrics.
   - The successful `trends-week` run included:
     - `pipeline.enrich.*`
     - `pipeline.analyze.*`
     - `pipeline.trends.*`
   - So the command was not “week trend only”; it was a multi-stage repair and
     synthesis job.

#### Why this is confusing

The CLI help text says:

- `trends`: “Generate trends for a period”
- `trends-week`: “Generate weekly trends and backfill missing daily trends”

That wording strongly implies a synthesis-only stage plus optional lower-level
trend backfill. It does not communicate “prepare + enrich + analyze + index +
trend synthesis, then recursively do the same for each missing day”.

#### Fixes

High priority:

- Expose `reuse_existing_corpus` at the CLI.
  - Example flags:
    - `--reuse-existing-corpus`
    - `--prepare-backlog/--no-prepare-backlog`
- Make the default behavior safer for week/month runs:
  - `trends-week` should probably default to reusing already analyzed corpus and
    only backfill missing day trend documents.

Command-surface improvement:

- Split the workflows explicitly:
  - `recoleta trends day`
  - `recoleta trends week`
  - `recoleta trends week --with-upstream`
  - `recoleta rerun-week --date ... --include ingest,analyze,trends,ideas`

Progress improvement:

- Emit stage heartbeats during long runs:
  - current stream
  - current day being backfilled
  - current substage: `prepare`, `analyze`, `index`, `trend_llm`
  - counts completed / total

### 3. `site serve` appeared to show only W11 weekly ideas, not each W11 day

#### Symptom

- During preview it looked like only the W11 weekly ideas were visible.

#### Root Cause

This is primarily a presentation issue, not a missing-output issue.

Evidence:

- The rebuilt site manifest contains the W11 daily idea pages under:
  - `site/en/ideas/...day--2026-03-09..15--ideas.html`
  - `site/zh-cn/ideas/...day--2026-03-09..15--ideas.html`
- The daily W11 idea markdown files also exist under both streams.

The likely source of confusion is the home-page preview logic:

- In `recoleta/site.py`, the home page renders only the latest `4` idea cards.
- Those latest cards are dominated by:
  - `2026-W12` weekly ideas
  - `2026-W11` weekly ideas
  - only the newest daily ideas after that

So if someone judged visibility from the home page, it is easy to conclude that
the week-level ideas are present while many day-level ideas are “missing”.

#### Fixes

UI improvement:

- Add an explicit note on the home page that it shows only the latest `4` idea
  briefs.
- Increase the home-page preview count, or make it separately show:
  - latest week ideas
  - latest day ideas

Navigation improvement:

- Add filters or tabs on `/ideas/index.html`:
  - `All`
  - `Week`
  - `Day`
  - per-stream filters

Docs improvement:

- Clarify that the home page is a preview, while `/ideas/` is the full archive.

### 4. The Chinese site did not include the new `2026W12` outputs

#### Symptom

- English site picked up the new W12 outputs.
- Chinese site still lagged behind.
- After rebuilding, manifest counts were:
  - English ideas: `18`
  - Chinese ideas: `16`

#### Root Cause

Localization is projection-based and separate from canonical trend/idea
generation.

- `recoleta trends` and `recoleta ideas` write canonical outputs.
- Localized trees are only produced by translation workflows.
- `site build` discovers localized language roots by scanning `Localized/...`
  directories; it does not translate on demand.

For this run:

- new canonical W12 trend and idea notes existed
- no matching W12 localized notes existed under `Localized/zh-cn/...`
- therefore the multilingual site build had nothing to include for those W12
  zh-CN pages

This is consistent with the design docs: localized notes are derived projections,
not automatically refreshed canonical outputs.

#### Fixes

High priority:

- Add an explicit post-generation workflow for localization:
  - `recoleta translate run --include trends,ideas`

Recommended UX improvement:

- Add a convenience option:
  - `recoleta ideas ... --translate`
  - or `recoleta trends-week ... --translate`
  - or `recoleta rerun-week ... --include translate`

Recommended site behavior:

- Consider optional fallback behavior for multilingual site builds:
  - show canonical content on the zh page with a “translation pending” badge
  - or omit only the zh localized detail page but keep the card visible with a
    fallback link to the canonical page

### 5. Other issues worth capturing

#### `--json` is not reliably machine-readable

- `trends-week --json` and `ideas --json` files were prefixed with LiteLLM
  “Provider List” output before the JSON object.
- `site build --json` also emitted log lines before the JSON payload.

Impact:

- downstream automation cannot safely parse stdout as strict JSON
- scripts must recover by scanning for the last JSON-looking line, which is brittle

Fix:

- force all logs and third-party console noise to stderr when `--json` is active
- add a regression test that validates stdout is a single JSON object with no prefix

#### Run success semantics are too optimistic

- The `trends-week` run finished as `succeeded`.
- The same run still recorded meaningful failure counts in enrich/analyze metrics.

Impact:

- operators cannot tell “clean success” from “partial success with silent fallout”

Fix:

- introduce richer terminal states such as:
  - `succeeded_clean`
  - `succeeded_with_partial_failures`
  - `failed`

#### Billing scope is muddied by hidden upstream work

- The run is labeled `command=trends-week`, but its billing includes upstream
  prepare/analyze/triage activity.

Impact:

- a user expecting “weekly trend cost” gets a mixed operational bill instead

Fix:

- separate billing by substage in the CLI summary
- or split upstream repair into an explicit command so weekly trend spend remains
  semantically narrow

## Recommended Follow-up

### High priority

- Normalize billing model aliases in analyze/translation/doctor.
- Expose `reuse_existing_corpus` or equivalent CLI control for trend commands.
- Stop `trends-week` from silently doing repeated `prepare + analyze` work by
  default.
- Add a one-shot rerun command that can include translation explicitly.
- Make `--json` stdout strict JSON only.

### Medium priority

- Improve `site` home/ideas UX so day vs week idea visibility is obvious.
- Add progress heartbeats for long-running trend backfill.
- Improve run terminal states and billing substage summaries.

### Nice to have

- Add a multilingual fallback mode for untranslated fresh outputs.
- Add a stream/day/week filter bar to the ideas archive.

## Why This Matters

The observed problems are connected.

- Billing looked wrong because the command surface hid upstream work and the
  analyze pricing path could not resolve the configured model alias.
- `trends-week` felt suspiciously slow because it was doing much more than its
  name suggests, and it did so with long silent gaps.
- Site preview confusion came from a mismatch between “home page preview” and
  “full archive”.
- Missing Chinese pages were not a site bug at all; they were the natural result
  of projection-based localization not being part of the run.

The common product need is better explicitness:

- explicit stage boundaries
- explicit billing semantics
- explicit localization workflow
- explicit preview/archive distinctions
