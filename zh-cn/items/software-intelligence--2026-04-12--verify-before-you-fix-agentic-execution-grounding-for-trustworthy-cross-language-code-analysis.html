Summary

摘要

这篇论文提出了一条跨语言漏洞分析流水线，可在 Java、Python 和 C++ 中检测、验证并修复代码。它的核心观点是，只有在执行确认漏洞可利用之后才应进行修复，这样可以减少流水线中的错误操作。

问题

  • 漏洞流水线常把模型预测当作事实，因此误报会流入后续阶段，浪费修复工作。
  • 现有工具通常把检测、验证和修复拆成彼此独立的系统，而且很多工具需要为每种编程语言单独使用不同模型。
  • 这很重要，因为基于未经验证的发现进行安全修复会产生不必要的补丁、降低信任，并错过真正的利用路径。

方法

  • 系统采用三阶段闭环：先检测可疑代码，再通过执行验证可利用性，最后只修复已确认的案例。
  • 在检测阶段，系统将 Java、Python 和 C++ 代码转换为统一的 Universal AST，并结合两类信号：程序结构的 GraphSAGE 嵌入，以及源码语义的 Qwen2.5-Coder-1.5B 嵌入。
  • 一个学习得到的双向门控机制会为每个样本分别给结构分支和语义分支赋权，这也能直接说明是哪一分支主导了预测。
  • 在验证阶段，LLM agent 会构建利用假设和测试 harness，在沙箱化的 Docker 环境中运行，并要求有确认性的执行证据后才将案例标记为可利用。
  • 在修复阶段，一个经过 LoRA 调优的 Qwen2.5-Coder-1.5B-Instruct 模型生成最小补丁，重新运行检测，并最多迭代五次；失败案例会连同诊断轨迹一起交给人工处理。

结果

  • 在同语言检测中，Java、Python 和 C++ 的准确率达到 89.84% 到 92.02%，F1 达到 0.8837 到 0.9109。
  • 在零样本跨语言检测中，六组训练-测试语言配对的 F1 达到 74.43% 到 80.12%；混合模型的平均 F1 为 0.7631，语义-only 为 0.6981，结构-only 为 0.5426。
  • 验证阶段对已标记样本确认可利用性的比例为 66.84% 到 71.49%，拒绝检测器误报的比例为 58.72% 到 62.37%，并以 2.67% 到 3.98% 的伪阳性率找回了 11.76% 到 16.21% 的漏检漏洞。
  • 修复成功率在五次迭代内达到 81.37% 到 87.27%，平均需要 2.3 到 3.4 次迭代；修复后的通过率为 90.44% 到 93.15%。
  • 从端到端看，整条流水线解决了 69.74% 的漏洞，消除了 61.24% 的误报，避免了 73.13% 的不必要修复，总体流水线失败率为 12.27%。
  • 消融实验表明两个关键组件都很重要：移除 uAST 会使平均跨语言 F1 下降 23.42%，禁用基于执行的验证会使不必要修复增加 131.7%，并使端到端成功率下降 9.56 个百分点。
