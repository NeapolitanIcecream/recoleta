Summary

摘要

SWE-chat 是一个大型数据集，收集了开源开发者在公开 GitHub 仓库中进行的真实编码代理会话。论文用它来衡量人们实际如何与编码代理协作、代理编写的代码有多少最终被保留，以及这些系统在日常开发工作流中会在哪些地方失败。

问题

  • 关于编码代理的研究很大程度上依赖精心构造的基准测试，但这些基准无法覆盖真实用户提示、反复往返的交互、工具使用，以及开发者最终实际保留了哪些代码。
  • 如果没有真实环境中的数据，就很难衡量实际效用、失败模式、人工监督、成本或安全影响。
  • 这件事之所以重要，是因为编码代理已经在大规模使用；如果产品或模型决策只依据基准结果，可能会误判其在真实工作流中的表现。

方法

  • 作者构建了 SWE-chat，这是一个持续更新的数据集，数据来自通过 Entire.io CLI 选择加入日志记录、并在公开 GitHub 仓库上工作的开发者。
  • 该数据集把完整会话转录与提交记录关联起来，包括用户提示、代理回复、工具调用、token 使用量、代码 diff，以及逐行的人类与代理作者归因。
  • 在论文写作时，SWE-chat 包含约 6,000 个会话、63,000 条用户提示、355,000 次代理工具调用、13,000 个检查点，以及分布在 200 多个仓库中的 270 万条日志事件。
  • 论文还增加了关于会话是否成功、用户画像、提示意图和用户反驳的标注，方法是结合经过验证的 LLM 评审器、原始日志，以及代码保留率、每已提交行成本、每已提交行耗时和基于 Semgrep 的安全发现等代码归因指标。
  • 核心思路很直接：端到端记录真实的人类-代理编码会话，然后比较代理产出了什么、开发者最终实际提交了什么，以及开发者在会话过程中如何回应。

结果

  • 在对比表中，SWE-chat 是第一个同时结合真实人类提示、代理工具轨迹、代码 diff和代码作者归因的数据集。
  • 编码行为呈现明显分化：22.7% 的会话完全由人类完成，36.5% 为协作模式，40.8% 为 vibe coding，即代理编写了至少 99% 的已提交代码。在三个月窗口内，vibe coding 在会话中的占比从 20% 增长到 超过 40%。
  • 使用场景不只包括补丁编写：最常见的具体提示意图是理解代码（19.0%）；创建新代码和git 操作各占 13.4%，调试占 13.0%。大约三分之一的代理工具调用是 bash 命令。
  • 代理编写的代码经常被丢弃：总体编码效率为 44.3%，代码保留率为 50.3%；在协作会话中，这两个数字分别降至 38.2% 和 44.1%，而 vibe coding 分别达到 59.0% 和 64.6%。
  • 按最终提交产出计算，vibe coding 的成本更高、速度更慢：中位数为每 100 行已提交代码消耗 204K tokens，约为协作模式的 3 倍；中位美元成本为每 100 行 0.13 美元，协作模式为 0.05 美元，纯人工为 0.07 美元；中位耗时为每 100 行 12.6 分钟，协作模式为 4.8 分钟，纯人工为 8.6 分钟。
  • vibe coding 的安全结果更差：按 Semgrep 测量，其引入的漏洞数为每 1,000 行已提交代码 0.76 个，协作模式为 0.14，纯人工为 0.08。论文还报告，用户经常会对代理输出提出反对：总体占所有轮次的 39%；其中用户打断约占 3.3% 到 6.0% 的轮次，而代理主动请求澄清只占 1.1% 到 2.6%。
