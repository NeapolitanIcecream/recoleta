Summary

本文提出一个面向LLM编程代理失败分析的可解释AI框架，把难以阅读的原始执行轨迹转成结构化、可视化、可操作的解释。核心价值是在真实调试场景中，帮助技术和非技术用户更快定位根因并提出更正确的修复方案。

Problem

  • 解决的问题：LLM编程代理经常失败，但失败时留下的是冗长、嵌套、难解释的执行日志，开发者和利益相关者很难看懂“哪里出错、为什么出错、怎么修”。
  • 为什么重要：如果失败原因不可解释，编程代理就难以被可靠部署到软件开发流程中，调试成本高，也难以建立用户信任。
  • 现有通用LLM解释不足：解释不一致、缺少领域结构、没有执行流程可视化、很少给出可直接落地的修复建议。

Approach

  • 先从32个真实失败案例中整理出一个编程代理失败分类体系（taxonomy），包括如 misunderstanding failure、iterative refinement failure 等类别；其中最常见的是“超过迭代上限但没有进展”。
  • 再用GPT-4/4.1 + structured outputs/function calling做自动标注：从原始轨迹中抽取错误信息、迭代次数、执行模式等特征，自动判定失败类别、子类别和置信度。
  • 然后构建一个混合解释系统：一部分生成执行流程图（Graphviz），一部分生成自然语言根因分析，另一部分给出反事实解释和修复建议。
  • 输出同时覆盖多种受众：既能生成适合人看的HTML报告，也能输出可集成进CI/CD或监控系统的JSON结果。
  • 最简单地说：方法就是“先把失败归类，再把日志画成流程，再用模板化+LLM生成解释和建议”，从而把杂乱日志变成可执行洞察。

Results

  • 数据与失败分布：在 87次 HumanEval 编程代理运行中，得到 32次失败、55次成功；失败中 18/32（56%） 属于 iterative refinement failure，是最常见模式。
  • 自动分类性能：在 32个失败样本上，自动分类达到 82.1%（26/32） 准确率；高置信预测（p>0.8）准确率 90.5%（19/21）；与人工标注的 Cohen’s κ = 0.76。
  • 用户研究规模：共 20人，其中技术用户 10人、非技术用户 10人，比较三种条件：原始轨迹、通用LLM解释、作者方法。
  • 理解速度：作者方法让用户比原始轨迹**快2.8×**理解失败；技术用户平均时间从 8.4±2.1 min 降到 3.0±0.8 min，非技术用户从 12.8±3.2 min 降到 4.2±1.1 min，且 p<0.01。
  • 根因识别与修复质量：技术用户根因识别从原始轨迹 42±15%、通用LLM 68±12% 提升到 89±8%；非技术用户从 18±12%、52±18% 提升到 76±11%。修复质量（1-5分）技术用户从 2.6±0.8 / 3.4±0.6 提升到 4.3±0.5，非技术用户从 1.4±0.6 / 2.8±0.7 提升到 3.8±0.6。
  • 置信度与定性反馈：技术用户理解置信度从 3.2±1.1 提升到 6.1±0.7，非技术用户从 2.1±0.9 提升到 5.6±0.8；18/20 参与者主动提到执行流程图“非常有帮助”或“必不可少”。
