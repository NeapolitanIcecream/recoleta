Summary

摘要

R²Eval 是一个基于真实 Python 项目构建的代码推理基准，而不是由简短的独立代码片段组成。它测试 LLM 在代码依赖真实代码仓库上下文和复杂数据类型时，能否正确预测程序输入和输出。

问题

  • 现有代码推理基准大多使用简短的生成代码片段或编程挑战题解，因此无法覆盖真实代码仓库的结构、依赖关系和大量对象参与的数据。
  • 许多以往基准把输入和输出限制为原始类型，这去掉了生产代码中一个常见的难点来源。
  • 这很重要，因为在简化基准上取得高分，可能会高估 LLM 在真实项目中执行程序修复、代码翻译和代码生成等软件任务时的表现。

方法

  • 论文提出了 R²Eval，这是一个包含 135 个代码推理问题的基准，数据来自 10 个常用 Python 项目：scikit-learn、django、requests、seaborn、sphinx、pytest、astropy、xarray、matplotlib 和 sympy。
  • 每个问题都是一个三元组 (P, I, O)：包含相关方法、依赖和类上下文的代码；序列化后的输入；以及序列化后的输出。
  • 核心机制是程序分析，它将复杂、复合和自定义的运行时对象拆解为可 JSON 序列化的部分，因此模型看到的是更接近真实情况的输入和输出，而不只是原始值。
  • 在评分时，该基准会把模型输出反序列化回对象，并通过运行测试检查正确性，从而避免仅靠字符串匹配带来的误判。
  • 评估将 R²Eval 与从 CRUXEval 中抽取的同规模 135 题样本进行比较，在输入预测和输出预测两项任务上测试六个 LLM。

结果

  • 在全部六个模型上，输入预测的平均准确率从 CRUXEval 上的 81.23% 降至 R²Eval 上的 16.91%，下降 64.32 个百分点。
  • 在全部六个模型上，输出预测的平均准确率从 80.37% 降至 28.15%，下降 52.22 个百分点。
  • o4-mini 在输入预测上从 92.59% 降至 20.00%，在输出预测上从 91.85% 降至 28.15%。
  • Gemini-2.5-Pro 在输入预测上从 91.85% 降至 15.56%，在输出预测上从 88.89% 降至 34.07%。
  • DeepSeek-R1 在输入预测上从 94.81% 降至 21.48%，在输出预测上从 87.41% 降至 31.85%。
  • 推理型模型的表现仍优于非推理型模型：在输入预测上平均高 13.95 个百分点，在输出预测上平均高 12.22 个百分点，但所有模型在这个更接近真实情况的基准上都明显退化。
