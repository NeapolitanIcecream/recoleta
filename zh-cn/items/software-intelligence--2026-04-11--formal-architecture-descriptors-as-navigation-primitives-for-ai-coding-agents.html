Summary

摘要

正式的架构描述符能帮助 AI 编码代理用更少的探索步骤找到正确的代码。论文认为，主要收益来自给代理提供明确的架构结构，而具体文件格式对模型理解的影响较小。

问题

  • AI 编码代理在修改代码前，会把大量工具调用耗费在代码库探索上，例如 grep、文件搜索和模块阅读。
  • 现有的上下文文件如 CLAUDE.md 或 AGENTS.md 往往较为非正式，而自动生成的仓库地图虽然能捕捉语法结构，却常常遗漏设计意图、约束和数据流。
  • 这很重要，因为额外的导航会增加成本，并可能降低任务成功率，尤其是在更大的代码库中，盲目搜索更困难。

方法

  • 论文提出了 intent.lisp，一种用嵌套 S 表达式编写的正式架构描述符，用来记录支柱、组件、符号、约束和数据流。
  • 一个 survey 工具会扫描仓库，并使用 LLM 起草该描述符；编码代理随后在任务中把这个描述符作为导航入口来读取。
  • 作者进行了三项研究：一项比较不同上下文格式的受控代码定位实验，一项使用无人工修订自动生成描述符的“产物 vs 过程”测试，以及一项覆盖 7,012 次 Claude Code 会话的观察性现场研究。
  • 他们还测试了写入端可靠性：要求 LLM 以 S 表达式、JSON、YAML 和 Markdown 生成描述符，再注入结构错误，以比较解析失败和静默损坏。
  • 论文对格式的主张比较收窄：S 表达式并不更容易被模型理解，但它更短，能通过语法强制层级结构，并且在生成出错时比 YAML 或 Markdown 更能保持可用。

结果

  • 在一个 2.2 万行 Rust 项目的 24 个代码定位任务中，使用 Claude Sonnet 4.6 时，架构上下文把平均导航步数从盲搜时的 5.2降到S 表达式下的 3.4、JSON 下的 3.4和Markdown 下的 2.9，降幅为 33–44%；盲搜与有上下文之间的差异具有显著性，S-expr 条件下 Wilcoxon p=0.009，效应量 d=0.92。
  • 在同一实验中，准确率的变化小于步数变化：盲搜 54%（13/24）、S 表达式 58%（14/24）、JSON 58%（14/24）、Markdown 63%（15/24）。另一项包含 20 个问题的理解测试发现，四种格式的准确率都是 95%，成对格式差异没有显著性。
  • 在一个 4.3 万行 Rust 项目上的 15 个任务中，“产物 vs 过程”研究显示，自动生成的 170 行描述符达到 100% 准确率，而盲搜为 80%，其中 p=0.002，d=1.04。这表明，即使没有开发者清理或代码重构，描述符文件本身也能提供帮助。
  • 在同一研究中，人工整理的 698 行描述符得到 87% 准确率和 2.9 步，而 AutoGen 为 3.9 步，盲搜为 5.6 步；AutoGen 的准确率高于 Curated，但 AutoGen-Curated 差异不显著（p=0.515）。
  • 在 96 次生成运行中，解析有效输出率分别为：JSON 100%、S 表达式 95.8%、YAML 91.7%。在 96 次错误注入中，静默损坏比例分别为：JSON 21%、S 表达式 50%、YAML 50%、Markdown 100%；S 表达式检测出了 100% 的结构完整性错误，而 YAML 对这类错误的检测率只有 50%。
  • 在 7,012 次 Claude Code 会话和 484,937 条消息中，引入正式描述符后，探索/编辑比值的 IQR 从 2.24 降到 1.08，即行为方差降低了 52%。这项现场研究是相关性研究，论文也指出，更大代码库和非定位类任务仍需直接测试。
