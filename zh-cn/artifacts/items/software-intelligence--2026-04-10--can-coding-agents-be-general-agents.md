# Can Coding Agents Be General Agents?

## Summary
## 摘要
这篇论文测试编码代理能否处理一般性的业务流程，而不只是软件任务。在一个 Odoo ERP 案例研究中，代理在简单任务上表现良好，但在更难的工作流上失效，尤其是在业务规则需要与代码执行保持一致时。

## 问题
- 当前基准把问题拆成两部分：SWE-Bench 和 Terminal-Bench 这类软件基准测试代码和系统能力，而 BFCL 和 $\tau$-bench 这类工具使用与策略基准简化了执行层。
- 真实的业务自动化需要同时具备这两方面能力：理解业务请求和策略，检查实时系统状态，决定应该发生什么，然后正确地修改软件状态。
- 论文提出的问题是，编码代理能否在一个真实的 ERP 系统中端到端完成这件事，因为用户已经在把它们用于软件工程之外的业务工作。

## 方法
- 作者将编码代理的泛化定义为业务层与代码层之间的双向转换：读取业务目标，用代码检查系统，规划业务动作，再把变更写回系统。
- 他们构建了一个案例研究环境，使用 **Odoo 19.0 Community Edition**，并填充了覆盖销售、库存、制造、采购和人力资源的真实公司数据。
- 任务是自然语言描述的业务工作流，并带有明确的约束和策略规则。每个场景都需要至少 **2 个相互依赖的决策** 和至少 **2 个相互作用的约束**；更难的场景需要 **5 个以上决策** 和 **5 个以上约束**。
- 代理只能使用 **bash 工具**、数据库凭据、一个工作区，以及五个 Odoo 数据模型的示例。它必须自行理解 ERP 并编写脚本。测试对象是将推理设置调到最高的 **GPT-5** 和 **Claude Sonnet 4.5**。
- 评估通过对照真实答案评分标准，检查最终 PostgreSQL 状态在 **约束解决、资源优化、可追溯性和策略遵循** 方面的表现。

## 结果
- 在 **10 个简单场景** 的首次试验中，Claude Sonnet 4.5 编码代理在验证器上的得分 **超过 80%**，并且能稳定完成创建销售订单、选择最低价供应商、生成发票等任务。
- 研究评估了 **20 个场景**，难度范围从 **Easy 到 Hardest**。随着约束数量增加，性能下降。
- 论文报告了一个定性上的模型差异：**GPT-5** 给出的业务方案往往与 Claude Sonnet 4.5 相当或更好，但它错误的 API 调用更多，因此总体分数更低。
- 在更复杂的场景中，代理从“方案可行但不够优”进一步退化为完全无法满足约束；随着工作流变得更复杂，可追溯性也随之下降。
- 论文识别出四种反复出现的失败模式：用于实现策略的 **懒惰代码启发式**、**业务层幻觉**、**忽略策略约束**，以及把成功的代码执行误认为正确业务结果的 **过度自信**。
- 摘录没有给出按模型和难度划分的完整数值评分表，因此最强的定量结论是：在 **10 个简单场景** 上达到 **80% 以上**，以及测试集包含 **20 个场景**，并且在更难任务上出现明显退化。
