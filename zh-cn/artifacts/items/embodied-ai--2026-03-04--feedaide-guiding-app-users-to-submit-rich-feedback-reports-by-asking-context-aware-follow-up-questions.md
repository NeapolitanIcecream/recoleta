# FeedAIde: Guiding App Users to Submit Rich Feedback Reports by Asking Context-Aware Follow-Up Questions

## Summary
FeedAIde 是一个面向移动应用反馈收集的上下文感知交互式系统，利用多模态大语言模型根据截图和交互上下文提出追问，帮助用户提交更完整、更有开发价值的反馈。论文展示了一个 iOS 框架实现，并在真实健身房员工应用中通过用户测试和专家评审验证其可用性与报告质量提升。

## Problem
- 移动应用用户反馈常常**模糊、缺少上下文、对开发者不够可操作**，导致开发者还要反复追问，增加沟通成本。
- 论文指出约 **45%** 的开发者对应用商店评论的回复是在索取更多细节，而很多用户**不会再回复**，或要等**数小时到数天**。
- 现有用户侧反馈支持方法没有把**截图、设备信息、交互日志**等上下文与 GenAI 结合起来，在提交当下即时引导用户补全关键信息。

## Approach
- 核心方法很简单：当用户触发反馈时，系统先自动收集**截图、设备信息、应用版本、交互日志**等上下文，再让 MLLM 猜测用户可能想报告什么。
- 系统先生成最多 **3** 个上下文相关的反馈候选，让用户选择或自行输入；随后固定进行 **2** 个简短的自适应追问，以补齐开发者最需要的信息。
- Prompt 设计分为三部分：**系统设定/约束**、**应用描述**、**动态上下文信息**，并强调只发送最相关上下文，避免 token 浪费、误导模型和隐私过度暴露。
- 最终模型输出结构化 JSON 报告，包含**用户意图摘要、开发者摘要、问答历史和上下文数据**，方便开发者直接处理。
- 作者实现了一个可复用的 **iOS Swift Package**，支持如 shake-to-report 的触发方式，并通过代理库接入不同 LLM 提供商。

## Results
- 在一个生产中的健身房员工 iOS 应用上进行了评估；**7 名**真实用户参加了组内实验，每人完成 **4** 个反馈场景（**2 个 bug 报告 + 2 个功能请求**）。
- 开发价值评估基于用户测试收集的 **54 份**报告，由 **2 名**行业专家进行质量评审；结论是 FeedAIde 对**bug 报告和功能请求**的质量都有提升，尤其是**完整性**更好。
- 与应用原有的简单文本反馈表单相比，参与者主观评价 FeedAIde **更容易使用、也更有帮助**；摘要未提供具体量表分数或显著性检验数值。
- 论文给出一个实际部署背景：该应用已使用**超过 1 年**、约有 **20 名**活跃用户，但此前仅收到 **2 条**问题报告，侧面说明原反馈机制摩擦较高。
- 定量结果的局限：在提供的节选中，**没有给出更细的数值指标**（如均值、标准差、p 值或具体质量评分差值），因此最强的具体主张是用户偏好提升和专家认定的报告完整性提升。
